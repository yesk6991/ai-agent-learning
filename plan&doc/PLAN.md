# AI Agent 学习项目 — 实现计划

## 项目目标

为一位正在转型 AI 工程化的前端工程师，构建一个**可运行、可学习、可扩展**的 Agent 系统，覆盖以下五大核心模块：

1. **Prompt Engineering** — 提示词技巧与模板系统
   - 为什么排第一：Prompt 是和 LLM 交互的唯一入口，所有 AI 应用的起点
   - ⚠️ 常见误区：Prompt ≠ 用户输入。用户输入只是 Prompt 的一小部分，Agent 发给 LLM 的是一大段精心构造的完整指令：
     - Prompt 模板 = React 组件（骨架固定），用户输入 = props（每次不同）：
       ```
       模板（固定骨架）：
         "你是一个知识库助手。参考资料：{{context}}  问题：{{question}}  如果资料中没有，请说不知道"

       用户问 "什么是 RAG？" → 搜索到文档 → 填入 context → LLM 基于文档回答
       用户问 "闭包是什么？" → 知识库没搜到 → context 为空 → LLM 说"不知道"
       → 同一个模板，不同填充 → 不同回答策略
       ```
     - "改 Prompt"改的是模板骨架和策略，不是每次填充的内容：
       ```
       改之前：模板里没有"请说不知道" → LLM 编造假答案
       改之后：模板里加了"如果资料中没有，请说不知道" → LLM 诚实回答
       → 这个改动对所有用户、所有问题都生效 → 所以需要评测确认整体效果
       ```
     ```
     ┌─────────────────────────────────────────────┐
     │  System Prompt（Agent 的"人设"）              │  ← Prompt 模块负责
     │  "你是一个知识库助手，优先使用知识库回答..."   │
     ├─────────────────────────────────────────────┤
     │  RAG 检索结果（自动注入的上下文）              │  ← Prompt 模块负责
     │  "参考资料：[1] RAG 是检索增强生成..."        │
     ├─────────────────────────────────────────────┤
     │  用户消息（用户输入的问题）                    │  ← 你说的"提示词"
     │  "什么是 RAG？"                              │
     └─────────────────────────────────────────────┘
              ↓ 三者拼在一起发给 LLM
     ```
     用户只输入了一句话，但 Agent 自动拼上了角色设定、RAG 检索结果、技巧增强等内容——**这就是 Prompt Engineering 的工程化价值**：不是解析用户输入，而是**构建发给 LLM 的完整指令**。
   - 你会学到：6 种核心 Prompt 技巧（Zero-shot → Few-shot → CoT → ReAct → 结构化输出 → 角色设定），以及如何用模板系统管理 Prompt 版本、做 A/B 测试
   - 面试价值：面试必考，CoT 和 ReAct 是 Agent 理论基础

2. **知识库 (RAG)** — 文档检索增强生成
   - 为什么重要：解决 LLM 的两大硬伤——知识过时和幻觉编造，企业 AI 落地的第一场景
   - ⚠️ 常见误区：LLM 不是已经读过全网了吗，为什么还需要知识库？——因为它们是两回事：
     ```
     LLM 训练数据里的知识（全网的、公开的）     RAG 知识库（你自己的、私有的）
     ─────────────────────────────        ─────────────────────────────
     "JavaScript 闭包是指..."              "公司 2026 年调薪方案如下..."
     "React Hooks 是 React 16.8 引入的"    "张三的绩效等级是 A，奖金系数 1.5"
     "余弦相似度计算公式是..."             "XX 项目的 API 接口文档..."
     ```
     LLM 的训练知识有四个硬伤：①有截止日期，之后的事不知道；②不知道你的内部信息（公司制度、客户数据）；③不知道时不会说"不知道"，而是编造假答案（幻觉）；④无法引用来源，无法验证。
     RAG 就是把你自己的文档"递给"LLM，让它基于真实文档回答，而不是凭记忆编造。
     > 类比：律师脑子里有法律知识（= LLM 训练知识），但具体到你的案件，他需要看你的合同（= RAG 知识库）。你不给合同，他只能凭经验猜——猜错了就是"幻觉"。
   - 你会学到：RAG 完整流水线（文档分块 → TF-IDF 向量化 → 余弦相似度检索 → 上下文注入 → LLM 生成），同时提供 LangChain 生产级方案对比
   - 面试价值：RAG 是 2024-2025 最火的 AI 工程化方向，懂原理+能实现=加分

3. **评测 (Evaluation)** — 输出质量评估与量化
   - 为什么不能跳过：没有评测，改了 Prompt 不知道变好还是变差，换模型不知道效果如何，整个 AI 开发就是"凭感觉"
   - ⚠️ 评测评的是什么：LLM 的输出质量——给同一个问题，你的 Agent 回答得对不对、全不全、格式对不对
   - 为什么需要评测 Prompt：因为改 Prompt 是 AI 开发中最频繁的操作，人肉判断"好不好"非常不可靠
     ```
     旧 Prompt："你是一个知识库助手"
     新 Prompt："你是一个知识库助手，如果知识库没有相关内容，请明确说不知道"

     没有评测 → "感觉好像好了一点？"（但可能有些问题反而回答更保守了）
     有了评测 → 跑 9 个用例，得分从 72 → 81，确认变好了 ✅
     ```
   - 什么时候会改 Prompt：在实际 AI 项目中几乎每天都在改——
     | 什么时候改 | 例子 | 不评测的风险 |
     |-----------|------|------------|
     | 用户反馈回答不对 | "问它 RAG 全称，它编了个假的" | 改完可能修了 A 问题，引入 B 问题 |
     | 换了模型 | Sonnet → Haiku 省钱 | Haiku 可能理解不了复杂的 Prompt |
     | 加了新功能 | 知识库新增文档，Prompt 要加检索结果注入 | 注入方式不对，模型可能忽略检索结果 |
     | 上线前验收 | "这个 Agent 能不能上生产？" | 没有量化标准，只能拍脑袋 |
     | A/B 测试 | 两个 Prompt 版本哪个好？ | 没有评测就无法做 A/B 对比 |
     > 类比：评测 = 前端的单元测试。你不会不写测试就重构代码，同样不该不跑评测就改 Prompt。
     > 评测不止开发阶段使用——三个阶段都会用到：
     > - 🛠️ 开发阶段（最频繁）：改 Prompt/模型/功能后跑评测确认没退化
     > - 🚀 上线前验收：全量用例跑一遍，分数达标才上生产
     > - 📊 线上监控：定期跑评测，发现分数下降→排查原因
   - 你会学到：4 类评测指标（字符串相似度/关键词命中率/JSON 格式检查/长度合规），评测运行器+报告生成，9 个预置评测用例
   - 面试价值：能说出"我的 Prompt 改了之后评测分数从 72 提升到 89"，比"我觉得效果好多了"有说服力 100 倍

4. **技能调用 (Tool Use)** — Agent 工具调用机制
   - 为什么是核心：没有工具调用的 LLM 只是"聊天机器人"，有了工具调用才是"Agent"——能自主决策、调用外部能力、完成复杂任务
   - ⚠️ "外部能力"是什么：就是开发人员写的代码函数。"外部"是相对于 LLM 而言的——LLM 只能生成文字，不能真正执行任何操作：
     ```
     LLM 能做的（只生成文字）          LLM 做不了的（需要工具=开发人员写的代码）
     ─────────────────────         ──────────────────────────────────
     "北京今天晴天，28°C"           查真实天气 → 调天气 API → 开发人员写的代码
     "3+5=8"（但不保证对）          精确计算 (3+5)*2 → 执行 JS 代码 → 开发人员写的代码
     "知识库中说RAG是..."          搜索知识库 → 跑检索算法 → 开发人员写的代码
     "现在是2026年6月"（可能错）    获取真实时间 → 读系统时钟 → 开发人员写的代码
     ```
     > 分工原则：LLM 负责决策（"该做什么"），工具负责执行（"具体怎么做"）。
     > 开发人员写：①工具代码 ②工具描述（告诉 LLM 有啥工具）③执行逻辑
     > LLM 做：理解问题 → 决定是否调工具 → 选哪个工具传什么参 → 根据结果生成回答
   - 你会学到：Agent Loop 引擎（用户提问 → 模型决策 → 工具执行 → 结果回传 → 模型生成），技能注册中心，4 个内置技能（天气/计算器/知识库搜索/时间），多轮工具调用
   - 面试价值：理解 ReAct 范式的工程实现，能讲清楚 Agent 和普通 Chatbot 的本质区别

5. **模型选型 (Model Selection)** — 模型对比与场景适配
   - 为什么需要：不是所有任务都用最强模型——翻译用 Haiku 就够了，用 Opus 是浪费钱；反过来复杂推理用 Haiku 会翻车
   - ⚠️ 怎么判断任务复杂还是简单：通过代码实现，有三种方式从简到复杂：
     ```
     方式 1：关键词匹配（本项目用的，学习原理）
       /翻译|translate/.test(text) → simple → Haiku
       /解释|explain/.test(text)  → medium → Sonnet
       /推理|reasoning/.test(text) → complex → Opus
       优点：零成本即时判断  缺点：用户不按套路出题就判断错

     方式 2：用便宜的小模型先分类（生产常用）
       用户提问 → 先发给 Haiku "判断复杂度" → Haiku 回答 "中等"
       → 再发给 Sonnet 处理
       优点：理解语义不怕换说法  缺点：多一次 API 调用

     方式 3：用户手动选择（最可靠）
       /model haiku   /model opus   /model auto
       优点：100% 准确  缺点：需要用户理解模型差异

     实际项目：方式 2+3 结合，默认自动选型，用户可手动覆盖
     ```
   - 你会学到：三级模型画像（Haiku 快+省 / Sonnet 均衡 / Opus 最强但贵），智能选型器（根据任务自动推荐），成本估算，A/B 对比实验
   - 面试价值：体现工程思维——不是"用最好的模型"，而是"用最合适的模型"

## 整体架构

> **学习顺序**：按文件名前的 ①②③... 顺序学习，这个顺序是按照依赖关系和认知递进设计的——先学基础再学依赖它的模块，每步都建立在前一步之上。

```
agent/
├── core/                        # 核心模块
│   │
│   ├── ① llm.js                 # 📡 LLM 调用层
│   │   作用：所有模块和 LLM 通信的唯一入口
│   │   内容：callLLM()（普通调用）、streamLLM()（流式调用）、Mock/Live 自动切换
│   │   为什么第一个学：其他所有模块都要通过它调用 LLM，是地基
│   │   运行：被其他模块内部调用，不需要单独运行
│   │
│   ├── prompt/                  # 🎯 Prompt Engineering
│   │   ├── ② techniques.js      #   技巧库
│   │   │   作用：6 种 Prompt 技巧的模板定义（Zero-shot/Few-shot/CoT/ReAct/结构化输出/角色设定）
│   │   │   内容：每种技巧的 apply() 方法 + 原理注释
│   │   │   为什么第二个学：Prompt 是 AI 应用的起点，理解了技巧才知道为什么要这样构造 Prompt
│   │   │   运行：npm run demo:prompt
│   │   │
│   │   └── ③ templates.js       #   模板系统
│   │       作用：将 Prompt 从"手写"升级为"模板+变量"，支持变量插值 {{var}} 和条件渲染
│   │       内容：5 个内置模板（通用问答/RAG问答/Agent系统/代码审查/评测打分）+ render() 函数
│   │       为什么紧接着学：技巧是"怎么写"，模板是"怎么管"——写好的 Prompt 需要模板化才能复用和 A/B 测试
│   │       运行：npm run demo:prompt（和 techniques.js 一起演示）
│   │

│   │   │   │   作用：存储分块后的文档及其向量，提供增删查接口
│   │   │   │   内容：基于数组的简单存储，支持 add/getAll/remove
│   │   │   │   为什么分块后学：有了分块结果才能存储，存储是检索的前提

│   │   │   │
│   │   │   ├── ⑥ embedder.js   #     TF-IDF 向量化

│   │   │   │   运行：npm run demo:rag

│   │   │   └── ⑦ retriever.js  #     余弦相似度检索

│   │   │       为什么最后学：它把分块+存储+向量化三个模块串联起来，是 RAG 的"出口"
```

## 模块详细设计

### 0. LLM 调用层 (llm.js) — 所有模块的底层依赖

这是整个 Agent 的"地基"——其他模块都要通过它和 LLM 通信，但你自己开发时不需要关心底层细节。

**核心设计：Mock/Live 自动切换**
- 有 API Key → 调用真实 Anthropic API（Live 模式）
- 无 API Key → 返回模拟数据（Mock 模式），Agent 流程照跑，只是回复是假的
- 为什么这么设计：让你在没有 API Key 时也能学习 Agent 的完整流程，不会被"调不了 API"卡住
- 渐进式解锁：配置 Key 后自动切换为 Live，代码不用改任何一行

**为什么用适配器模式：**
```
上层模块（Agent Loop / RAG / 评测）都调用 callLLM()
                    ↓
            callLLM() 内部判断：
              有 Key → Anthropic API
              无 Key → Mock 数据
                    ↓
上层代码无需关心底层是真实还是模拟——这就是"适配器模式"
```

**还提供流式调用接口 streamLLM()：**
- 支持逐字输出（像 ChatGPT 打字效果）
- Agent 的对话模式用 streamLLM()，评测用 callLLM()

### 1. Prompt Engineering 模块

### 1. Prompt Engineering 模块

**techniques.js** — 6 种核心技巧，每种都有原理注释 + 可运行示例：
- **Zero-shot**：直接提问，观察模型基准表现。最简单的起点，也是评测时的 Baseline
- **Few-shot**：给出 2-3 个示例，引导输出格式。核心发现：示例比指令更有效——模型更擅长"模仿"而非"理解规则"
- **Chain-of-Thought (CoT)**："一步步思考"，推理链。GSM8K 基准测试中，CoT 让数学推理正确率从 17% 提升到 80%+，是目前最可靠的提分技巧
- **ReAct**：Reason + Act 交替，Agent 基础范式。让模型先推理"我需要什么信息"，再行动"调用工具获取"，循环直到问题解决。这是所有 Agent 框架（LangChain/AutoGPT/CrewAI）的理论基础
- **结构化输出**：JSON Schema 约束输出格式。API 集成场景必须——你的代码需要解析模型输出，自由文本无法可靠解析
- **角色设定**：System Prompt 定义 Agent 人格。所有 Agent 的起点——先定义"你是谁"，再定义"你能做什么"

**templates.js** — Prompt 模板系统：
- 变量插值 `{{variable}}`：同一模板填不同参数生成不同 Prompt
- 条件渲染 `{{#if variable}}...{{/if}}`：根据场景动态增减 Prompt 内容
- 5 个内置模板：通用问答、RAG 问答、Agent 系统、代码审查、评测打分
- 便于 A/B 测试：同一问题用不同模板，评测分数对比，用数据说话而非"我觉得"

### 2. 知识库 (RAG) 模块

**本地实现（核心学习路径）—— 零依赖，纯 JS，原理一目了然**：
- `chunker.js`：按固定大小 / 语义段落分块，带重叠（overlap）
  - 为什么需要分块：LLM 上下文窗口有限（128K tokens），不能把整个文档塞进去；小块语义更聚焦，检索更精准
  - 为什么需要 overlap：避免关键信息恰好在切割点被截断，overlap = chunk_size × 10%~20%
- `embedder.js`：TF-IDF 向量化（纯 JS 实现，无需外部 API）
  - 向量化的本质：把文本变成数字，让"语义相近的文本在数字空间中距离也近"
  - TF-IDF 的局限：基于词频，不理解语义（"开心"和"快乐"被视为无关），适合学习原理
  - 生产方案：OpenAI text-embedding-3-small（语义级理解，但需要 API 调用）
- `retriever.js`：余弦相似度检索 Top-K
  - 余弦相似度：衡量两个向量方向的相似性，值域 [-1, 1]，越接近 1 越相似
  - Top-K 检索：取出最相关的 K 个文档片段，通常 K=3~5
- `store.js`：内存向量存储，支持 CRUD
  - 全量扫描 O(n) 适合学习；生产环境用 ANN 近似最近邻索引 O(log n)
- 完整流程：文档加载 → 分块 → 向量化 → 索引 → 查询 → 上下文注入 → LLM 生成

**LangChain 实现（生产级对比）**：
- `pipeline.js`：使用 LangChain.js + OpenAI Embeddings + Chroma/Pinecone
- 需额外依赖，单独 README 说明安装步骤
- 与本地版对比，理解工程化 RAG 的取舍：本地版理解原理，LangChain 版理解工程实践

### 3. 评测 (Evaluation) 模块

**metrics.js** — 4 类评测指标：
- **字符串相似度**：基于 Levenshtein 编辑距离，值域 [0, 1]。适合评测翻译、总结等输出与参考答案接近的任务
- **关键词命中率**：预期关键词是否出现在输出中。适合评测事实性问答——问"RAG 全称"，关键词 ["Retrieval", "Augmented", "Generation"] 出现几个
- **JSON 格式检查**：输出是否为合法 JSON，是否包含必需字段。适合评测结构化输出任务
- **长度合规**：输出是否在预期长度范围内。太短可能不完整，太长可能有冗余
- 综合得分：各指标平均，0~1 分

**runner.js** — 评测运行器：
- 批量运行评测用例，逐条打印得分
- 汇总评分，生成报告：总体通过率、平均分、按分类/难度细分
- 支持对比不同 Prompt/模型的评测结果——改了 Prompt 后跑一次评测，分数涨了就是真提升

**cases.js** — 9 个预置评测用例：
- Prompt 类（3 个）：零样本问答、CoT 推理、结构化输出
- RAG 类（3 个）：基础知识检索、分块策略、向量化
- Agent 类（3 个）：工具调用、计算器调用、RAG+工具综合
- 每个用例包含：输入问题、预期关键词、难度等级

### 4. 技能调用 (Tool Use) 模块

**registry.js** — 技能注册中心（注册表模式）：
- 统一注册/发现技能，新增技能只需：1) 写定义文件 2) 在这里注册
- 自动生成 Anthropic Tool Schema 数组——模型通过 Schema 知道有哪些工具可用
- 运行时根据模型返回的 tool_use 名称，找到对应执行函数

**executor.js** — Agent Loop 引擎（核心中的核心）：
- 传统 API 调用是单向的：User → LLM → Response
- Agent 是循环的：User → LLM → Tool Call → Execute → LLM → Tool Call → ... → Final Response
- 这个循环让 Agent 具备了"自主行动"的能力——模型自己决定要不要调工具、调哪个工具
- 安全措施：MAX_ITERATIONS = 5 防止无限循环
- 与 RAG 模块串联：`search_knowledge_base` 技能让 Agent 在对话中自主决定是否查知识库

**技能定义**（每个技能 = Anthropic Schema + 执行函数）：
- `weather.js`：天气查询（模拟数据）。最经典的 Tool Use 示例，面试必问
- `calculator.js`：数学计算（真实执行）。LLM 不擅长精确计算，这是已知局限——通过工具"委托"给代码
- `search.js`：知识库搜索（调用 RAG 模块）。体现 Agent "按需行动"——需要知识时自己查，不需要时不查
- `timer.js`：时间查询。Agent 需要感知时间来做出合理决策

### 5. 模型选型 模块

**profiles.js** — 模型画像数据库（三维度评估：能力/成本/延迟）：
- Claude Haiku 4.5：通用 6 分，$0.8/M 输入，延迟 0.5s → 适合翻译/分类/提取等简单任务
- Claude Sonnet 4.6：通用 8 分，$3/M 输入，延迟 1s → 适合对话/写作/分析/Agent/RAG 等大多数任务
- Claude Opus 4.8：通用 10 分，$15/M 输入，延迟 3s → 适合复杂推理/规划/架构设计
- 成本差距：Opus 是 Sonnet 的 5 倍，Sonnet 是 Haiku 的 ~4 倍——选错模型=烧钱

**selector.js** — 智能模型选择器：
- 任务检测：关键词匹配判断简单/中等/复杂任务
- 自动推荐：简单→Haiku，中等→Sonnet，复杂→Opus
- 约束调整：预算有限自动降级，延迟要求高倾向快模型
- 给出推荐理由——不是黑盒推荐，而是可解释的

**comparator.js** — A/B 对比实验：
- 同一问题发给不同模型，对比输出质量、Token 消耗、延迟
- 用数据说话，而不是"我觉得这个模型更好"

### 6. Agent 主逻辑 (agent.js)

将五大模块串联为完整 Agent：
1. 接收用户输入
2. Prompt 模板渲染（用模板系统构造 System Prompt）
3. 模型选择器决定使用哪个模型（根据任务自动选型）
4. 如果需要知识，调用 RAG 检索（检索结果注入 System Prompt）
5. 模型决定是否调用工具（Agent Loop）
6. 工具执行，结果回传
7. 生成最终回答
8. 评测模块可选择性评分

### 7. CLI 交互 (cli.js)

- 基于 readline 的命令行交互
- 支持命令：`/help` `/rag on|off` `/model <name>` `/eval` `/demo` `/quit`
- 彩色输出，Token 用量实时显示

## API Key 处理策略

由于用户暂无 API Key，项目将：
1. **所有模块支持 Mock 模式**：无 Key 时使用模拟数据运行，仍可学习流程
2. **.env 配置指导**：首次运行时提示获取和配置 API Key
3. **渐进式解锁**：有 Key 后功能逐步解锁（Mock → 真实 API）

## 实现顺序

1. 先搭建项目骨架和 Mock 基础设施
2. Prompt 模块（最简单，立即可用）
3. 技能调用模块（在现有 tools.js 基础上扩展）

5. 评测模块（可对前三个模块的输出打分）

7. Agent 主逻辑（串联所有模块）
8. LangChain RAG 实现（对比学习）
9. CLI 交互界面
